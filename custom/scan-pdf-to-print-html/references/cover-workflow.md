# Cover Workflow: HTML Cover → Handout

The handout concept-map cover is now an **HTML cover**, not an SVG. The cover is generated by the `a4-novak-html-cover` skill (HTML cards + SVG connector overlay + KaTeX), rendered to a full-A4-page **`concept-map.png`**, and auto-injected into `handout.html` by the postprocess step. This keeps the cover editable at its source (the `.html` file) while requiring zero cover-specific logic in the handout builder.

This replaces the legacy `<img src="concept-map.svg">` default. The old SVG path still works as a fallback for jobs that never migrated.

## Why HTML, not SVG

A pure-SVG cover needs hand-tuned coordinates for every card and pre-rendered MathJax path data for every formula — each edit risks coordinate drift, black-fill fallback, or line-over-text. The HTML流派 (validated A/B against the 数列 cover) makes editing a card = changing text, and lets `drawEdges` compute connector geometry from live DOM. The trade-off accepted here: the cover reaches the handout as a rendered PNG (formulas not selectable inside the handout), chosen over inlining because inlining the cover's HTML would pollute the handout's global CSS/JS (`.card`, `.formula`, `id="map"`, `* {}` reset, KaTeX double-load). Editability is preserved one level up, in `concept-map.html`.

## End-to-end flow

1. **Generate the cover** with `a4-novak-html-cover`. It produces, in a cover job dir:
   - `concept-map.html` — the editable source (cards + `EDGES` config + KaTeX + drawEdges/autoFitFormula engine).
   - `concept-map.png` — the **full-A4-page render** (210mm × 297mm aspect). This is the handout-consumption artifact.
   - `concept-map.pdf` — optional standalone preview.
   - `vendor/katex/` — local KaTeX used by the HTML source.

2. **Place `concept-map.png` in the handout job dir** (next to `handout.html`). You may also copy the editable `concept-map.html` for future revisions; only the `.png` is consumed.

3. **Run the postprocess step** (Markdown-Source Mode step 3):
   ```
   py -3 scripts/postprocess_handout_for_contract.py --html handout.html
   ```
   `inject_cover_metadata` scans the job dir for cover candidates and writes the matched filename into `data-cover-href` on the print root. Then `injectConceptCover` (JS) creates the first sheet as `<article class="sheet concept-map-sheet">` containing `<img class="concept-map-front-image" src="concept-map.png">`. No manual `<img>` placement is needed or wanted.

4. **Render** the handout (step 4). The cover PNG fills the first A4 sheet via the `.concept-map-sheet` CSS (`width/height: 100%; object-fit: fill`).

## Candidate priority (PNG-first)

`inject_cover_metadata` checks candidates in this order:

```python
("concept-map.png", "concept-map.svg", "concept-map-preview.png")
```

PNG is first so HTML-cover jobs win. Consequences:

- **HTML-cover job:** ship `concept-map.png`. Do **not** leave a stale `concept-map.svg` in the same dir — if the `.png` is later deleted, the handout would silently fall back to an outdated SVG.
- **Legacy job:** an existing `concept-map.svg` with no `.png` still works (SVG is the fallback).
- **Mixed dir (png + svg both present):** PNG wins. This is intended for migration; if you intentionally want the SVG, remove the PNG.

## Revising a cover

To change a cover that is already feeding a handout:

1. Edit `concept-map.html` (card text / a `data-tex` formula / an `EDGES` entry / layout).
2. Re-run `py -3 render_cover.py concept-map.html` to regenerate `concept-map.png`.
3. Re-run the handout postprocess + render. Do not edit the PNG directly.

## Validation

- The handout's `validate_rendered_handout_contract.py` checks cover-vs-regular sheet alignment in screen preview (`.concept-map-sheet` must share the A4 frame `left`/`width` of regular sheets).
- The `a4-novak-html-cover` skill's `validate_cover_formulas.py` should pass on the cover HTML **before** rendering the PNG (catches formula overflow at the source).
- Regression eval `svg-cover-injection` (eval #2) covers the injection mechanism with an SVG fixture; it still passes because the mechanism is filename-driven and the eval's job dir has no PNG, so it falls back to the SVG fixture.

## Migration of old jobs

Old jobs with `concept-map.svg` are not broken. To modernize a cover, regenerate it with `a4-novak-html-cover` (analyze the same document → new `concept-map.html` + `concept-map.png`), drop the old `.svg` into the job dir, and re-run postprocess. Do not hand-edit the legacy SVG.
